# Invoices

Brief intro: LatticeStripe provides full Invoice lifecycle management — from creating draft invoices to collecting payment. This guide walks through the common workflows, collection methods, and pitfalls.

Link to Stripe docs: [Stripe Invoice API](https://docs.stripe.com/api/invoices)

## The Invoice Workflow

The canonical manual invoice flow:

1. Create a draft invoice with `auto_advance: false`
2. Add line items via `InvoiceItem.create/3`
3. Finalize the invoice with `Invoice.finalize/4`
4. Collect payment with `Invoice.pay/4`

Code example showing all 4 steps with a client.

## Collection Methods

Two collection methods:
- `:charge_automatically` — Stripe charges the customer's default payment method automatically
- `:send_invoice` — Stripe emails the invoice; customer pays via hosted page

Code example for each. Note: `send_invoice` requires `days_until_due`.

## Auto-Advance Behavior

**This is the most common footgun with Stripe invoices.**

When `auto_advance` is omitted (or `true`), Stripe auto-finalizes draft invoices in ~1 hour. This means if you create a draft, add items over the next 2 hours, the invoice may already be finalized before you're done.

LatticeStripe emits a telemetry warning when `Invoice.create/3` is called without explicit `auto_advance`:

```elixir
# If you use attach_default_logger, you'll see:
# [warning] Invoice created without explicit auto_advance — Stripe will auto-finalize in ~1 hour. Set auto_advance: false for draft invoices.
LatticeStripe.Telemetry.attach_default_logger()
```

**Recommendation:** Always pass `auto_advance: false` for draft invoices you plan to modify.

## Working with Invoice Items

InvoiceItems are standalone CRUD resources attached to draft invoices before finalization.

Code example: create an InvoiceItem, list items on the invoice.

**Key distinctions:**
- `InvoiceItem` (standalone CRUD) — what you create to add charges
- `Invoice.LineItem` (read-only) — what appears on the rendered invoice

Note the 250-item limit per invoice.

`price` vs `price_data`: You can reference an existing Price by ID or pass inline price_data.

## Draft Invoice Management

Draft invoices support full mutation:
- `Invoice.update/4` — modify metadata, description, etc.
- `Invoice.delete/3` — permanently delete (draft only!)
- `InvoiceItem.update/4` — modify line items
- `InvoiceItem.delete/3` — remove line items

Once finalized, these operations fail.

## Action Verbs

After finalization, drive the invoice through its lifecycle:
- `Invoice.finalize/4` — draft -> open
- `Invoice.pay/4` — open -> paid
- `Invoice.void/4` — open -> void
- `Invoice.mark_uncollectible/4` — open -> uncollectible
- `Invoice.send_invoice/4` — email the invoice to the customer

Code examples for each. Note state preconditions.

## Proration Preview

Preview what the next invoice will look like before confirming a subscription change:

```elixir
{:ok, preview} = LatticeStripe.Invoice.upcoming(client, %{
  "customer" => "cus_xxx",
  "subscription" => "sub_xxx",
  "subscription_items" => [%{"id" => "si_xxx", "price" => "price_new"}],
  "subscription_proration_behavior" => "create_prorations"
})

# preview.id is nil — this is a preview, not a real invoice
# preview.lines contains the projected line items
```

Note: Stripe is deprecating `upcoming` in favor of `create_preview`. Use `create_preview/3` for new integrations.

For strict proration mode:
```elixir
client = LatticeStripe.Client.new!(
  api_key: "sk_test_...",
  finch: MyApp.Finch,
  require_explicit_proration: true
)

# This will return {:error, %Error{type: :proration_required}}
# because proration_behavior is not specified:
LatticeStripe.Invoice.upcoming(client, %{"customer" => "cus_xxx"})
```

## Subscription-Generated Invoices

Stripe automatically creates invoices for subscriptions. The `billing_reason` field tells you why:
- `:subscription_cycle` — regular billing period
- `:subscription_create` — initial subscription invoice
- `:subscription_update` — mid-cycle proration
- `:manual` — created via API

These invoices follow the same lifecycle but are created automatically.

## Testing Invoices with Test Clocks

Use Billing Test Clocks (Phase 13) to test invoice behavior over time:

Code example showing test clock advancement through `days_until_due`.

## Common Pitfalls

1. **Auto-advance surprise** — Always set `auto_advance: false` for draft invoices
2. **Missing items** — Create InvoiceItems before finalizing; you can't add items after
3. **send_invoice without days_until_due** — Required for `collection_method: :send_invoice`
4. **Deleting finalized invoices** — `delete/3` only works on drafts; void finalized ones
5. **Stale struct status** — Don't check `invoice.status` before calling action verbs; Stripe is the authority
