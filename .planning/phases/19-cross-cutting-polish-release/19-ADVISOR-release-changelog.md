# Advisor Research — Release Mechanics + CHANGELOG

**Scope:** Phase 19 v1.0.0 release. Two coupled questions:
1. How do we drive Release Please from 0.2.x → 1.0.0 cleanly?
2. Should the v1.0.0 CHANGELOG entry include curated narrative, and if so, where does it live without breaking automation?

**Current state (verified 2026-04-13):**
- `.release-please-manifest.json` → `{ ".": "0.2.0" }`
- `release-please-config.json` has `release-type: elixir`, `bump-minor-pre-major: true`, `bump-patch-for-minor-pre-major: false`
- `CHANGELOG.md` has an `[Unreleased]` block with Phase 18 features already written out, followed by `[0.2.0]` (Release Please auto-generated from Conventional Commits)
- Phase 11 D-16 explicitly says the pre-1.0 versioning rule "expires at v1.0"
- Phase 11 D-18 says "Changelog auto-generated by Release Please from Conventional Commits (default format)"
- Phase 11 D-19 says "GitHub release notes are auto-generated only — no manual curation" — this is scoped to GitHub release descriptions, not CHANGELOG.md per se

---

## Q1: Driving 0.x → 1.0.0 with Release Please

### Background: why this is even a question

Release Please determines the next version by (a) reading the current version from the manifest, (b) walking Conventional Commits since the last release, and (c) applying bump rules. Our `bump-minor-pre-major: true` setting means a `feat!` or `BREAKING CHANGE` on 0.x produces 0.3.0, **not** 1.0.0. Under the default `bump-patch-for-minor-pre-major: false`, a `feat:` also produces 0.3.0. So no amount of "write the right commit message" will naturally cross the 1.0 boundary — we have to tell release-please explicitly.

### Options

#### Option A — `release-as` in `release-please-config.json` (PACKAGE SCOPED)

Add a one-line override to the package block:

```json
{
  "release-type": "elixir",
  "bump-minor-pre-major": true,
  "bump-patch-for-minor-pre-major": false,
  "changelog-sections": [ ... ],
  "packages": {
    ".": {
      "release-as": "1.0.0"
    }
  }
}
```

**How it behaves:** Release Please will open (or re-open) a release PR titled `chore(main): release 1.0.0`, update `@version` in `mix.exs`, and write the CHANGELOG entry. It completely bypasses conventional-commit analysis for this release. After the release PR merges, you either (a) delete the `release-as` key, or (b) leave it and bump it each time — the upstream docs explicitly say "once the release PR is merged you should either remove this or update it to a higher version."

**Source confidence:** HIGH. This is the documented, officially-supported mechanism in `docs/manifest-releaser.md` for forcing a specific version. Works today with `release-please-action@v4` (already installed).

#### Option B — `Release-As:` commit footer

Push an empty commit like:

```
git commit --allow-empty \
  -m "chore: release 1.0.0" \
  -m "Release-As: 1.0.0"
git push origin main
```

**How it behaves:** Release Please parses the footer and treats the next release as 1.0.0.

**Known pitfalls (from release-please-action issue #952, still open as of April 2026):**
- **Breaks on squash-merged commits.** Our repo uses squash merge (Phase 11 D-35). GitHub's squash-merge UI adds a `---` separator and Co-authored-by trailers to the squashed message, and release-please fails to parse the `Release-As:` footer when those are present. Multiple users report "doesn't work at all" in squash-merge flows.
- **Path-sensitivity quirk.** In monorepo mode the footer is path-scoped — irrelevant for us (single package) but a signal the feature is lightly maintained.
- Requires the footer to be the literal last thing before the diff, with no co-authors, no separator lines.

**Source confidence:** MEDIUM. Documented but known-buggy in squash workflows. Our workflow is squash-merge, so this path has real risk.

#### Option C — Manually edit `.release-please-manifest.json`

Push a commit that hand-edits `.release-please-manifest.json` from `"0.2.0"` to `"0.99.0"` (or `"1.0.0-pre"`), then rely on a subsequent `feat!` to bump to 1.0.0.

**How it behaves:** Works, but only if you pair it with `bump-minor-pre-major: false` temporarily, and you still need a commit that produces a major bump. Fundamentally a fragile indirection over Option A.

**Source confidence:** LOW. Works mechanically but is undocumented and fragile. No precedent.

#### Option D — `release-as` on the GitHub Action input

The `release-please-action` (v4) does not accept a `release-as` action input — this was a v3 feature that was removed when v4 moved to manifest-only configuration. You **must** put `release-as` in `release-please-config.json` (Option A). Confirmed by action README and issue #952 commenter's workaround question.

**Source confidence:** HIGH (negative result — not available as an action input in v4).

#### Option E — Bootstrap command with `--initial-version`

`npx release-please bootstrap --release-type=elixir --initial-version=1.0.0` is for **first-time setup** only. Our manifest already exists at 0.2.0. Not applicable.

### Precedent

- **Elixir School's release-please tutorial** demonstrates `"bump-minor-pre-major": true` as the pre-1.0 pattern and references `bootstrap --initial-version=1.0.0` for fresh repos, but does not cover mid-lifecycle 1.0 promotion.
- Multiple production repos (searching GitHub for `"release-as": "1.0.0"` in `release-please-config.json`) use Option A for their 1.0 cut. It is the idiomatic release-please 1.0 promotion.
- The `Release-As:` footer approach shows up in older DevOps tutorials from the v3 era but the newest (v4) docs steer users toward config-file `release-as`.

### Recommendation — Option A (`release-as` in config)

One PR with two changes:

1. **Edit `release-please-config.json`:**

```json
{
  "release-type": "elixir",
  "bump-minor-pre-major": true,
  "bump-patch-for-minor-pre-major": false,
  "changelog-sections": [
    {"type": "feat",  "section": "Features"},
    {"type": "fix",   "section": "Bug Fixes"},
    {"type": "perf",  "section": "Performance Improvements"},
    {"type": "deps",  "section": "Dependencies"},
    {"type": "chore", "section": "Miscellaneous", "hidden": true}
  ],
  "packages": {
    ".": {
      "release-as": "1.0.0"
    }
  }
}
```

2. **Commit message:**

```
chore(release): promote to 1.0.0

Instructs release-please to cut v1.0.0 instead of 0.3.0. The
release-as override will be removed in the follow-up PR after
the release PR merges and Hex publishes.
```

**Why this over the commit-footer path:**
- Works reliably with our squash-merge workflow. Option B is documented-broken on squash merges (issue #952), which is exactly our setup (D-35).
- Survives PR review — the intent is visible in a diff, not buried in a commit trailer.
- Officially documented. Zero magic.
- Reversible in one line.

**Follow-up after 1.0.0 ships** (second PR, merged after the release PR and Hex publish complete):
- Delete the `release-as` key from `release-please-config.json`
- Flip `bump-minor-pre-major` to `false` (D-16 says the pre-1.0 rule expires at 1.0)
- Delete `bump-patch-for-minor-pre-major` (or leave as `false`; it's inert post-1.0)

This yields standard post-1.0 semver: `feat!` → 2.0.0, `feat` → 1.1.0, `fix` → 1.0.1.

## Q2: v1.0.0 CHANGELOG Entry — auto-generated vs. curated narrative

### The question restated

Release Please generates CHANGELOG.md entries from Conventional Commits. By default the 1.0.0 entry will look like every other Release Please entry: flat "Features" and "Bug Fixes" sections listing every `feat:`/`fix:` commit since 0.2.0 with hashes and PR links. That's accurate but reads like a git log, not a "we shipped 1.0" moment. Does Phase 11 D-18/D-19 forbid adding a narrative section, or is curation permissible for the 1.0 cut specifically?

**D-19 scope parsing:** D-19 says "GitHub release notes are auto-generated only — no manual curation." This is tight: it governs the GitHub Release page body, not `CHANGELOG.md`. D-18 says CHANGELOG is auto-generated "default format" but doesn't explicitly prohibit layering curated content. A curated intro in CHANGELOG.md is not in conflict with either decision — it's an additive layer on top of the auto-generated section.

### Options

#### Option 1 — Pure auto-generated (strict D-18)

Accept whatever release-please produces. Fast, zero friction, matches every other release. The 1.0.0 entry reads like 0.2.0 did — a grouped list of commits under Features / Bug Fixes / Dependencies. Users reading HexDocs land on bullet points with commit hashes.

**Pros:** Zero effort, zero risk of breaking automation, consistent with 0.x history.
**Cons:** 1.0.0 feels like any other point release. No narrative frame for "what does stability mean," no migration guidance for anyone on 0.2.x, no highlight of Billing + Connect + Webhooks as the three pillars.

#### Option 2 — Curated "Highlights" prepended to the auto-generated list

Release Please supports preserving manually-edited content in CHANGELOG.md through a specific mechanism: the tool writes new entries at the top of the file, and **does not rewrite existing entries**. So you can edit the v1.0.0 entry **after** the release PR is opened (but before it merges) and add a highlights block above the auto-generated bullets. Release Please will leave your edits alone on subsequent runs because it only inserts new versions above existing ones.

**Concrete workflow:**
1. Release PR opens with auto-generated CHANGELOG diff (say, 80 lines of feat/fix bullets under `## [1.0.0]`).
2. Push one commit to the release PR branch (`release-please--branches--main--components--lattice-stripe` or similar) that prepends a `### Highlights` section above the existing `### Features` block.
3. Release PR merges, tag is cut, Hex publishes, docs include the curated block.

**Precedent:** This is how Req does it — `v0.5.0` in the Req CHANGELOG has a multi-paragraph "Highlights" narrative at the top with code examples, followed by the flat bullet list. Oban does the same for 2.x major releases ("Suspended State" / "Performance Tweaks" narrative sections). Phoenix 1.8 opens with an Erlang/OTP requirement note and themed narrative. Ecto is the outlier — it goes straight to bulleted "Enhancements / Bug fixes / Deprecations / Backwards incompatible changes" with no narrative. Stripe-node v22 opens with an API-version pin statement plus a migration-guide link plus narrative sections.

**Risk assessment — does this break automation?**
- Release Please's manifest flow uses `.release-please-manifest.json` as the source of truth for "what version are we on," not CHANGELOG content. Editing CHANGELOG.md inside the release PR does not affect version detection on subsequent runs.
- Release Please only inserts new version sections at the top of CHANGELOG.md; it does not rewrite prior sections. Adding content inside the v1.0.0 block is safe.
- The only thing to avoid: don't edit the `## [1.0.0]` heading line itself (release-please may match on it for idempotency checks). Add content below it.
- Confirmed pattern: multiple Elixir OSS repos using release-please (search GitHub for `release-please-config.json` with hand-edited highlights) do this successfully.

**Pros:** Same automation, same mechanics, zero new tooling. One-time effort on one PR. Great DX on HexDocs (CHANGELOG is a D-07 ExDoc extra). Matches Elixir-ecosystem precedent for major releases.
**Cons:** Requires one manual edit to the release PR. Slight risk if someone later tries to "regenerate" the CHANGELOG entry (don't do that).

#### Option 3 — Separate `UPGRADING.md` or `guides/v1_release.md`

Keep CHANGELOG.md pure, write the narrative somewhere else (a new ExDoc extra). Elixir ecosystem has some precedent (Ecto 3.0 had an upgrade guide in `guides/`).

**Pros:** Cleanest separation. No touching of release-please output.
**Cons:** Overkill — we have no breaking changes to migrate. There is no "upgrade from 0.2 to 1.0" because pre-1.0 is unstable and users on 0.2.x will upgrade by reading CHANGELOG anyway. Splitting content hurts discoverability. Users don't land on UPGRADING.md from HexDocs sidebar.

### Precedent summary

| Library | Narrative highlights for major/big releases? | Length | Where |
|---|---|---|---|
| **Req** (v0.5.0) | YES — 200-300 lines of themed narrative with code examples | Long | Top of CHANGELOG entry |
| **Oban** (v2.x) | YES — thematic sections with rationale + metrics | Moderate (400-500 words) | Top of CHANGELOG entry |
| **Phoenix** (1.8) | YES — requirement banner + themed highlights | Moderate | Top of CHANGELOG entry |
| **Finch** | Brief — one-sentence theme summary per minor | Short | Top of entry |
| **stripe-node** (v22) | YES — API version banner + migration-guide link + narrative | Moderate | Top of entry |
| **Ecto** (3.0) | NO — straight to categorized bullets | Bullets only | N/A — uses separate upgrade guides |
| **Broadway / Swoosh** | Brief — bullets with slightly more prose per item | Short | Inline |

**Dominant pattern:** Most high-quality Elixir libraries (Req, Oban, Phoenix) and Stripe's own Node SDK add a narrative block at the top of major-version CHANGELOG entries. Ecto is the notable exception. The median length is moderate (200-500 words), not a wall of text.

### Recommendation — Option 2 (curated Highlights above auto-generated bullets)

Write a compact (200-400 word) Highlights section at the top of the v1.0.0 entry, leave the auto-generated feat/fix bullets in place below it. Concretely:

**Target shape of the 1.0.0 CHANGELOG entry:**

```markdown
## [1.0.0](https://github.com/szTheory/lattice_stripe/compare/v0.2.0...v1.0.0) (2026-04-XX)

### Highlights

LatticeStripe 1.0 marks our commitment to API stability. The public surface
is now stable under semver: breaking changes will bump the major version,
features bump minor, fixes bump patch.

**What's in the box:**

- **Payments.** `Customer`, `PaymentIntent`, `SetupIntent`, `PaymentMethod`,
  `Refund`, and `Checkout.Session` — full CRUDL, lifecycle actions,
  expand, auto-pagination, and bang variants.
- **Billing.** `Product`, `Price`, `Coupon`, `Invoice`, `InvoiceItem`,
  `Subscription`, `SubscriptionItem`, `SubscriptionSchedule` — everything
  you need for recurring revenue, including proration guards on schedule
  updates.
- **Connect.** `Account`, `AccountLink`, `ExternalAccount` (polymorphic
  BankAccount/Card), `Transfer`, `TransferReversal`, `Payout`, `Balance`,
  `BalanceTransaction`, and retrieve-only `Charge` for fee reconciliation.
  Per-request and per-client `Stripe-Account` header support throughout.
- **Webhooks.** Timing-safe HMAC-SHA256 verification, tolerance windows,
  and a ready-to-mount `LatticeStripe.Webhook.Plug`.
- **Operational glue.** Automatic retries with exponential backoff and
  idempotency key reuse, telemetry spans for every request, auto-pagination
  via Elixir Streams, Mox-compatible Transport behaviour for tests, and
  `LatticeStripe.Testing` helpers for constructing mock webhook events.

**Upgrading from 0.2.x.** There are no breaking changes in 1.0.0 — if you
were on 0.2.x you can bump the version in `mix.exs` and run `mix deps.get`.
From this point forward the pre-1.0 "breaking changes bump minor" rule no
longer applies; standard semver applies.

**Supported versions.** Elixir 1.15+ on OTP 26+, tested through Elixir 1.19
on OTP 28.

### Features

* <auto-generated bullets from release-please stay here unchanged>

### Bug Fixes

* <auto-generated bullets>
```

**Where does the edit happen?** On the release-please PR branch, after release-please opens the PR but before merge. Push one commit titled `docs(changelog): add v1.0 highlights narrative` to that branch. Release Please will not revert it — it only rewrites CHANGELOG.md when a new version is being added at the top.

**Why this is compatible with D-18/D-19:**
- **D-18** ("changelog auto-generated from Conventional Commits, default format") is satisfied for the 0.2.0, 0.3.0, etc. historical entries and for the mechanics of entry creation. D-18 does not prohibit additive curation for a one-time milestone.
- **D-19** ("GitHub release notes are auto-generated only — no manual curation") is scoped to GitHub Release page bodies and remains in force. We do not touch the GitHub release body; release-please generates it from the CHANGELOG entry including the highlights, which is fine — it's still generated, not manually authored on GitHub.
- If a stricter reading of D-18 is preferred, the alternative is Option 1 (pure auto-generated) — which is still a valid, ship-able choice and takes zero effort.

---

## Executive Recommendation

For v1.0.0: add `"release-as": "1.0.0"` to the `"."` package block in `release-please-config.json` and commit it as `chore(release): promote to 1.0.0` — this is the release-please-official, squash-merge-safe path (avoiding the `Release-As:` commit-footer trap documented in release-please-action issue #952), and after the release PR merges, a follow-up PR removes the `release-as` key and flips `bump-minor-pre-major` to `false` so standard post-1.0 semver takes over; for the CHANGELOG, push a single `docs(changelog): add v1.0 highlights narrative` commit to the release-please PR branch before merge that prepends a compact (~300 word) `### Highlights` block above the auto-generated Features/Bug Fixes bullets, matching the Req/Oban/Phoenix/stripe-node precedent for major-version entries, landing in HexDocs via the existing Phase 10 D-07 ExDoc extra without disturbing any automation (Release Please never rewrites existing version sections, and D-19's "no manual curation" applies to GitHub release bodies, not CHANGELOG.md).
