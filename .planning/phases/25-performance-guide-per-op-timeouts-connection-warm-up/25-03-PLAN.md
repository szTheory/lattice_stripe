# Performance

LatticeStripe is designed for production Stripe integrations. This guide covers

Include these exact elements in each section:

**1. Pool Sizing:**
- Explain that Stripe uses HTTP/1.1, so each Finch pool worker holds 1 persistent TCP connection
- Explain `size` (concurrent connections per pool) and `count` (number of parallel pools)
- Throughput formula: `Max concurrent Stripe requests = size * count`
- Three concrete pool configs with labeled use cases:
  - Conservative (early-stage SaaS, <100 req/s): `size: 10, count: 1` = 10 concurrent
  - Standard production (moderate traffic, 100-500 req/s): `size: 25, count: 2` = 50 concurrent
  - High-throughput (>500 req/s, batch/webhook): `size: 50, count: 4` = 200 concurrent
- Each config as a complete Elixir code block with `{Finch, name: MyApp.Finch, pools: %{...}}`
- Note: these are starting points — use `Finch.get_pool_status/2` to observe actual saturation

**2. Supervision Tree:**
- Complete `Application.start/2` example showing:
  - Finch child spec with production pool config
  - Client creation after supervisor start
  - warm_up/1 call with graceful error handling (Logger.warning, don't crash)
- Use the exact pattern from RESEARCH.md Code Examples section
- Include `require Logger` for the warning call

**3. Per-Operation Timeouts:**
- Explain the three-tier precedence: per-request opts[:timeout] > operation_timeouts[op_type] > client.timeout
- Show the exact example from success criteria: `operation_timeouts: %{list: 60_000, search: 45_000}`
- Show a complete Client.new!/1 call with operation_timeouts
- Explain valid keys: `:list`, `:search`, `:create`, `:retrieve`, `:update`, `:delete`
- Explain that `nil` default preserves existing 30s behavior — zero behavior change for existing callers
- Recommended values table:
  | Operation | Default (ms) | Recommended for heavy workloads |
  |-----------|-------------|--------------------------------|
  | :list     | 30_000      | 60_000                         |
  | :search   | 30_000      | 45_000                         |
  | :create   | 30_000      | 15_000                         |
  | :retrieve | 30_000      | 10_000                         |
  | :update   | 30_000      | 15_000                         |
  | :delete   | 30_000      | 15_000                         |

**4. Connection Warm-Up:**
- Explain what "warm" means: TLS handshake + HTTP connection pre-established; subsequent requests skip the ~100-300ms handshake latency
- Show `LatticeStripe.warm_up/1` in `Application.start/2` (from section 2, but expanded)
- Show `LatticeStripe.warm_up!/1` for strict startup where failure should crash
- Explain why the function sends `GET /v1/` (lightweight, no side effects, establishes connection)
- Explain return values: `{:ok, :warmed}` on any HTTP response (even 404), `{:error, reason}` on transport failure
- Cross-reference to [Client Configuration](client-configuration.html) for full client setup

**5. Benchmarking:**
- Show how to enable pool metrics: `start_pool_metrics?: true` in Finch pool config
- Show `Finch.get_pool_status/2` usage with `Enum.each` to print pool utilization
- Explain what to look for: `in_use_connections` near `pool_size` means the pool is saturated — increase `size` or `count`
- Cross-reference to [Telemetry](telemetry.html) for request-level timing via telemetry events

**6. Common Pitfalls:**
- **Single-pool bottleneck**: Using default Finch config (no `pools:` key) gives a small default pool. Always configure explicit pool sizing for Stripe.
- **Not warming up**: First request after deploy takes ~100-300ms extra for TLS. Call `warm_up/1` in `Application.start/2`.
- **Overly aggressive timeouts on list/search**: Stripe list endpoints scan large datasets. Setting `timeout: 5_000` globally will cause failures on lists with many pages. Use `operation_timeouts` to give list/search more room.
- **Ignoring pool saturation**: Monitor with `Finch.get_pool_status/2`. If connections are always saturated, increase pool size before adding more application instances.
