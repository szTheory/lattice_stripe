<decisions>
## Implementation Decisions

### CI Workflow Structure
- **D-01:** Single `ci.yml` workflow file with separate jobs: lint, test matrix, integration test. All jobs run in parallel (no `needs:` dependencies between them).
- **D-02:** Triggers on `pull_request` (any branch) and `push` to `main` only. Path filtering: skip CI on docs-only changes (.md, .planning/, guides/).
- **D-03:** Test matrix: 3 combos — Elixir 1.15/OTP 26 (floor), 1.17/OTP 27 (middle), 1.19/OTP 28 (latest). Fail-fast strategy.
- **D-04:** Separate lint job on latest Elixir/OTP runs: `mix format --check-formatted`, `mix compile --warnings-as-errors`, `mix credo --strict`, `mix docs`, `mix hex.build` (smoke test), `mix deps.audit` (MixAudit).
- **D-05:** Separate integration test job with stripe-mock as GitHub Actions service container (ports 12111-12112). Runs `mix test --include integration` on latest Elixir/OTP only.
- **D-06:** Cache deps/ and _build/ keyed on mix.lock hash. No Docker image caching (stripe-mock is small, pull fresh each run for latest OpenAPI spec).
- **D-07:** Concurrency groups with cancel-in-progress: `concurrency: { group: ${{ github.workflow }}-${{ github.ref }}, cancel-in-progress: true }`.
- **D-08:** Minimal `permissions: { contents: read }` at workflow level. Escalate only for specific jobs that need more.
- **D-09:** Use `erlef/setup-beam@v1` for Elixir/OTP installation. Pin all GitHub Actions to major tags (e.g., `actions/checkout@v4`).
- **D-10:** Run on `ubuntu-latest`. Job timeout: 15 minutes.
- **D-11:** Workflow-level env block: `MIX_ENV: test`. Integration job adds `STRIPE_MOCK_HOST: localhost`, `STRIPE_MOCK_PORT: 12111`.
- **D-12:** No test partitioning (suite is small enough). No coverage tracking. No artifact uploads. GitHub default notifications only.

### Release & Versioning
- **D-13:** Separate `release.yml` workflow for Release Please + Hex publishing. Triggered on push to main (for Release Please) and release created (for Hex publish).
- **D-14:** Release Please configured via manifest (`release-please-config.json` + `.release-please-manifest.json`). Release type: `elixir`. Auto-updates `@version` in mix.exs.
- **D-15:** Fully automatic Hex publishing — when Release Please PR merges and creates a GitHub release, `mix hex.publish --yes` runs automatically. HEX_API_KEY stored as GitHub repo secret, rotated annually.
- **D-16:** Version strategy: start at 0.1.0 (current), semver-ish pre-1.0 (breaking changes bump minor, features bump patch).
- **D-17:** Git tag format: v-prefixed (v0.1.0, v1.0.0). Matches existing `source_ref: "v#{@version}"` in mix.exs.
- **D-18:** Changelog auto-generated by Release Please from Conventional Commits (default format). Already listed as ExDoc extra.
- **D-19:** GitHub release notes are auto-generated only — no manual curation.
- **D-20:** No pre-publish CI re-run — rely on branch protection having validated the Release Please PR.
- **D-21:** `mix hex.publish` also publishes docs to HexDocs automatically (default behavior).

### Hex Package Metadata
- **D-22:** Standard package() metadata in mix.exs: name, description, licenses: ["MIT"], links: %{GitHub, Changelog, HexDocs}, files: ["lib", "mix.exs", "README.md", "CHANGELOG.md", "LICENSE"]. Public package (no Hex organization).

### Dependency Management (Dependabot)
- **D-23:** Dependabot checks weekly on Mondays. Two ecosystems: `mix` (Hex dependencies) and `github-actions`.
- **D-24:** Dev-only dependencies grouped into single PRs. Production dependencies get individual PRs.
- **D-25:** Auto-merge enabled for patch version updates only. Minor and major require manual review.
- **D-26:** Max 5 open Dependabot PRs at a time.
- **D-27:** Conventional Commits prefix: `commit-message: { prefix: "chore" }`. Produces messages like `chore(deps): bump finch from 0.21.0 to 0.22.0`.
- **D-28:** Labels: `dependencies`. No reviewers, no assignees, no milestones.
- **D-29:** No ignore rules — monitor all dependencies, all update scopes (patch, minor, major).
- **D-30:** Security updates enabled (GitHub default). Combined with MixAudit in CI for defense in depth.
- **D-31:** Auto-rebase enabled (Dependabot default). Target branch: main.
- **D-32:** No dependency vendoring. mix.lock committed to repo.
- **D-33:** Manual Hex version retirement when needed (`mix hex.retire`). No automation.
- **D-34:** MixAudit runs in CI only (lint job) — no separate scheduled audit.

### Repo & Branch Policy
- **D-35:** Squash merge strategy for all PRs. Keeps main history linear and clean.
- **D-36:** Branch protection on main: require all CI status checks to pass, prevent force pushes. Allow maintainer bypass for emergencies. No signed commit requirement.
- **D-37:** Auto-delete head branches after merge.
- **D-38:** README badges: CI status, Hex version, HexDocs, License.
- **D-39:** Create CONTRIBUTING.md with: dev setup, running tests (including stripe-mock), Conventional Commits format, PR process, branch naming convention (feat/, fix/, chore/, docs/).
- **D-40:** GitHub issue templates (bug report, feature request) and PR template with checklist.
- **D-41:** Create MIT LICENSE file (currently missing).
- **D-42:** Create SECURITY.md with: vulnerability reporting process (email, not public issue), expected response time, supported versions.
- **D-43:** GitHub repo settings: enable Issues, disable Wiki (use HexDocs), disable Discussions, add topics (elixir, stripe, payments, sdk).
- **D-44:** No CODEOWNERS file. No stale issue/PR bot.
- **D-45:** .gitignore is sufficient as-is — no changes needed.
