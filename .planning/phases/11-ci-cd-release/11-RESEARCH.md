# Phase 11: CI/CD & Release - Research

**Researched:** 2026-04-03
**Domain:** GitHub Actions CI, Release Please, Hex publishing, Dependabot
**Confidence:** HIGH

## Summary

Phase 11 delivers the complete automation layer for LatticeStripe: GitHub Actions CI across three Elixir/OTP combos, Release Please for semver versioning, automated Hex publishing, and Dependabot dependency management. All major decisions are locked in CONTEXT.md — research confirms that the chosen toolchain (erlef/setup-beam, googleapis/release-please-action@v4, Dependabot with groups) is current and correct for 2026.

The key implementation risk is the paths-ignore + branch protection interaction: when a PR touches only docs files, path filtering causes the workflow to be skipped entirely, which leaves required status checks in "Pending" state and blocks merge. The locked decision (D-02) uses paths-ignore on both push and pull_request — the planner must decide whether to use a "skip" job pattern or accept that docs-only PRs require a manual bypass. This is flagged as an open question.

The stripe-mock service container, cache keying with OTP/Elixir in the key, Release Please manifest config, and Dependabot auto-merge via `dependabot/fetch-metadata` + `gh pr merge --auto` are all well-understood with verified patterns.

**Primary recommendation:** Use `erlef/setup-beam@v1` for Beam setup, `googleapis/release-please-action@v4` with `command: manifest` for releases, and a GitHub Actions workflow (not native Dependabot auto-merge) for patch auto-merge since it enables the required status-check gate.

### Pattern 1: ci.yml Structure

Three parallel jobs: `lint`, `test`, `integration`. No `needs:` between them (all start simultaneously per D-01).

```yaml
# Source: verified from erlef/setup-beam README + fly.io Elixir CI guide
name: CI

on:
  push:
    branches: [main]
    paths-ignore:
      - "**.md"
      - ".planning/**"
      - "guides/**"
  pull_request:
    paths-ignore:
      - "**.md"
      - ".planning/**"
      - "guides/**"

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

env:
  MIX_ENV: test

permissions:
  contents: read

jobs:
  lint:
    name: Lint
    runs-on: ubuntu-latest
    timeout-minutes: 15
    steps:
      - uses: actions/checkout@v4
      - uses: erlef/setup-beam@v1
        with:
          elixir-version: '1.19'
          otp-version: '28'
      - uses: actions/cache@v3
        with:
          path: deps
          key: ${{ runner.os }}-1.19-28-mix-${{ hashFiles('**/mix.lock') }}
          restore-keys: ${{ runner.os }}-1.19-28-mix-
      - uses: actions/cache@v3
        with:
          path: _build
          key: ${{ runner.os }}-1.19-28-build-${{ hashFiles('**/mix.lock') }}
          restore-keys: ${{ runner.os }}-1.19-28-build-
      - run: mix deps.get
      - run: mix format --check-formatted
      - run: mix compile --warnings-as-errors
      - run: mix credo --strict
      - run: mix docs
      - run: mix hex.build
      - run: mix deps.audit

  test:
    name: Test (Elixir ${{ matrix.elixir }} / OTP ${{ matrix.otp }})
    runs-on: ubuntu-latest
    timeout-minutes: 15
    strategy:
      fail-fast: true
      matrix:
        include:
          - elixir: '1.15'
            otp: '26'
          - elixir: '1.17'
            otp: '27'
          - elixir: '1.19'
            otp: '28'
    steps:
      - uses: actions/checkout@v4
      - uses: erlef/setup-beam@v1
        with:
          elixir-version: ${{ matrix.elixir }}
          otp-version: ${{ matrix.otp }}
      - uses: actions/cache@v3
        with:
          path: deps
          key: ${{ runner.os }}-${{ matrix.elixir }}-${{ matrix.otp }}-mix-${{ hashFiles('**/mix.lock') }}
          restore-keys: ${{ runner.os }}-${{ matrix.elixir }}-${{ matrix.otp }}-mix-
      - uses: actions/cache@v3
        with:
          path: _build
          key: ${{ runner.os }}-${{ matrix.elixir }}-${{ matrix.otp }}-build-${{ hashFiles('**/mix.lock') }}
          restore-keys: ${{ runner.os }}-${{ matrix.elixir }}-${{ matrix.otp }}-build-
      - run: mix deps.get
      - run: mix test

  integration:
    name: Integration Tests
    runs-on: ubuntu-latest
    timeout-minutes: 15
    services:
      stripe-mock:
        image: stripe/stripe-mock:latest
        ports: ['12111:12111', '12112:12112']
    env:
      STRIPE_MOCK_HOST: localhost
      STRIPE_MOCK_PORT: '12111'
    steps:
      - uses: actions/checkout@v4
      - uses: erlef/setup-beam@v1
        with:
          elixir-version: '1.19'
          otp-version: '28'
      - uses: actions/cache@v3
        with:
          path: deps
          key: ${{ runner.os }}-1.19-28-mix-${{ hashFiles('**/mix.lock') }}
          restore-keys: ${{ runner.os }}-1.19-28-mix-
      - uses: actions/cache@v3
        with:
          path: _build
          key: ${{ runner.os }}-1.19-28-build-${{ hashFiles('**/mix.lock') }}
          restore-keys: ${{ runner.os }}-1.19-28-build-
      - run: mix deps.get
      - run: mix test --include integration
```

**Cache key insight:** Include Elixir + OTP versions in the `_build` cache key. The compiled artifacts are version-specific — reusing `_build` from a different OTP version causes mysterious compile errors. `deps/` is less sensitive but including versions avoids edge cases too.

**Service container insight:** When running directly on the ubuntu runner (not inside a container), service containers are accessible at `localhost` on their mapped ports. No wait/health-check step needed for stripe-mock — it starts fast and GitHub Actions waits for ports to be ready automatically.

### Pattern 2: release.yml Structure

Two jobs with output threading: `release-please` runs on every push to main and outputs `release_created`; `publish-hex` runs only when `release_created == 'true'`.

```yaml
# Source: elixirschool.com/blog/managing-releases-with-release-please (verified)
name: Release

on:
  push:
    branches: [main]

permissions:
  contents: write
  pull-requests: write

jobs:
  release-please:
    name: Release Please
    runs-on: ubuntu-latest
    outputs:
      release_created: ${{ steps.release.outputs.release_created }}
      tag_name: ${{ steps.release.outputs.tag_name }}
    steps:
      - uses: googleapis/release-please-action@v4
        id: release
        with:
          command: manifest
          token: ${{ secrets.GITHUB_TOKEN }}

  publish-hex:
    name: Publish to Hex.pm
    runs-on: ubuntu-latest
    needs: release-please
    if: ${{ needs.release-please.outputs.release_created == 'true' }}
    steps:
      - uses: actions/checkout@v4
      - uses: erlef/setup-beam@v1
        with:
          elixir-version: '1.19'
          otp-version: '28'
      - run: mix deps.get
      - run: mix hex.publish --yes
        env:
          HEX_API_KEY: ${{ secrets.HEX_API_KEY }}
```

**Permissions note:** The `release-please` job needs `contents: write` and `pull-requests: write` to create/update PRs and create releases. These elevated permissions are only in `release.yml`, not `ci.yml` (which keeps `contents: read` per D-08).

### Pattern 3: Release Please Manifest Config

Two files in repo root:

**release-please-config.json:**
```json
{
  "release-type": "elixir",
  "bump-minor-pre-major": true,
  "bump-patch-for-minor-pre-major": false,
  "changelog-sections": [
    {"type": "feat", "section": "Features"},
    {"type": "fix", "section": "Bug Fixes"},
    {"type": "perf", "section": "Performance Improvements"},
    {"type": "deps", "section": "Dependencies"},
    {"type": "chore", "section": "Miscellaneous", "hidden": true}
  ]
}
```

**`bump-minor-pre-major: true`** means `feat:` commits bump minor (0.1.0 → 0.2.0) before 1.0.0 is reached — aligns with D-16 (breaking changes bump minor pre-1.0).

**`bump-patch-for-minor-pre-major: false`** means `feat:` still bumps minor, not just patch.

**.release-please-manifest.json:**
```json
{
  ".": "0.1.0"
}
```

The `"."` key is the repo root. This must be set to the current deployed version (0.1.0 for initial release).

**How it updates mix.exs:** The `elixir` release type knows to update the `@version "x.y.z"` line in mix.exs. No custom configuration needed.

### Pattern 4: Dependabot Configuration

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "mix"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
    open-pull-requests-limit: 5
    labels:
      - "dependencies"
    commit-message:
      prefix: "chore"
      include: "scope"
    groups:
      dev-dependencies:
        dependency-type: "development"
        patterns: ["*"]
    # No ignore rules per D-29

  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
    open-pull-requests-limit: 5
    labels:
      - "dependencies"
    commit-message:
      prefix: "chore"
      include: "scope"
```

**`include: "scope"` behavior:** Produces commit messages like `chore(deps): bump finch from 0.21.0 to 0.22.0` — the `(deps)` scope comes from the `include: "scope"` directive. This aligns with D-27.

**Dev dependency grouping:** The `groups` with `dependency-type: "development"` groups all dev/test deps (mox, ex_doc, credo, mix_audit) into a single PR per D-24. Production deps get individual PRs (no group defined for them).

### Pattern 5: Dependabot Auto-Merge Workflow

A separate workflow triggered on `pull_request` that uses `dependabot/fetch-metadata` to check update type:

```yaml
# .github/workflows/dependabot-automerge.yml  (or add as job to ci.yml)
name: Dependabot Auto-Merge

on: pull_request

permissions:
  contents: write
  pull-requests: write

jobs:
  auto-merge:
    runs-on: ubuntu-latest
    if: github.actor == 'dependabot[bot]'
    steps:
      - name: Dependabot metadata
        id: metadata
        uses: dependabot/fetch-metadata@v2
        with:
          github-token: "${{ secrets.GITHUB_TOKEN }}"
      - name: Enable auto-merge for patch updates
        if: steps.metadata.outputs.update-type == 'version-update:semver-patch'
        run: gh pr merge --auto --squash "$PR_URL"
        env:
          PR_URL: ${{ github.event.pull_request.html_url }}
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

**Why GitHub Actions workflow (not native Dependabot auto-merge):** The workflow approach gates on CI checks passing (via `--auto` flag) before merging. Native Dependabot auto-merge has similar behavior but the workflow approach gives explicit audit log entries and is more commonly documented.

**`--squash`:** Matches the repo's squash merge strategy (D-35).

### Anti-Patterns to Avoid

- **Single cache entry for all matrix combos:** Using only `hashFiles('mix.lock')` without OTP/Elixir version in the key causes `_build` artifacts from one OTP version to be reused for a different one. This causes cryptic BEAM version mismatch errors. Always include matrix values in the cache key.
- **Sharing elevated permissions across all jobs:** Only the `release.yml` workflow needs `contents: write`. `ci.yml` should stay at `contents: read`.
- **Using `stripe/stripe-mock` version-pinned tags:** The point of stripe-mock is to test against the latest Stripe OpenAPI spec. Using `latest` Docker tag is correct and intentional (D-06).
- **Running `mix hex.publish` without `--yes`:** Interactive confirmation will hang CI. Always pass `--yes`.
- **Release Please `command: manifest` without the two config files:** The `manifest` command requires both `release-please-config.json` and `.release-please-manifest.json` to exist. Without them the action errors.
- **Omitting the `name:` field in `package()`:** Without `name:`, the Hex package name defaults to the app atom (`:lattice_stripe`) which is correct here, but being explicit avoids confusion.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Changelog generation | Custom changelog script | Release Please | Parses conventional commits, handles sections, deduplication, linking — extremely tedious to reproduce correctly |
| Version bumping from commits | Custom git log parser | Release Please `elixir` release type | The tool knows Elixir's `@version` pattern and handles semver pre-1.0 correctly |
| Dependabot PR merge logic | Custom webhook + merge bot | `dependabot/fetch-metadata` + `gh pr merge --auto` | Official action with proper metadata parsing; `--auto` gates on CI |
| Beam version installation | Custom install scripts | `erlef/setup-beam@v1` | Handles binary caching, version resolution, PATH setup across OTP/Elixir cross-versions |
| Stripe API mock server | Local stub module | `stripe/stripe-mock:latest` Docker | Official, auto-updated from Stripe OpenAPI spec; validates request shapes, not just responses |

## Common Pitfalls

### Pitfall 1: paths-ignore + Branch Protection Deadlock

**What goes wrong:** When a PR touches ONLY files in the `paths-ignore` list (all `.md`, `.planning/`, `guides/`), GitHub skips the CI workflow entirely. The required status checks that branch protection requires (lint, test, integration) never run and stay in "Pending" — the PR cannot be merged.

**Why it happens:** GitHub does not auto-pass skipped checks. A skipped workflow is different from a passed workflow.

**How to avoid:** Two options:
1. **Accept it as intentional:** For a library where docs-only changes to main are rare and maintainers can bypass (D-36 allows bypass), this is acceptable. Docs PRs just need a maintainer bypass.
2. **Skip job pattern:** Instead of `paths-ignore` at the workflow `on:` level, add a step in each job that detects "is this docs-only?" and uses `if: ${{ !steps.check.outputs.docs_only }}` with a check job that always reports success. This is more complex but fully solves the problem.

**Warning signs:** A docs PR sits in "Waiting for required checks" state that never resolve.

**Recommendation:** Given D-36 allows maintainer bypass, Option 1 (accept + bypass) is the simpler choice for a solo-maintained OSS library. Document in CONTRIBUTING.md that docs-only PRs need maintainer bypass.

---

### Pitfall 2: Cache Key Does Not Include OTP/Elixir Version

**What goes wrong:** The `_build` directory contains BEAM bytecode compiled for a specific OTP version. If the cache key only uses `mix.lock` hash, a test matrix job on Elixir 1.15/OTP 26 might restore `_build` compiled for Elixir 1.19/OTP 28. This causes `FunctionClauseError` or `UndefinedFunctionError` at runtime, which looks like test failures but is actually a build artifact issue.

**Why it happens:** `hashFiles('**/mix.lock')` is the same across all matrix combos since they use the same deps.

**How to avoid:** Always include matrix values in the cache key:
```
key: ${{ runner.os }}-${{ matrix.elixir }}-${{ matrix.otp }}-build-${{ hashFiles('**/mix.lock') }}
```

**Warning signs:** Tests fail on one matrix combo but pass on others with errors that don't make sense for the test code.

---

### Pitfall 3: Release Please Runs Before mix.exs Has a package() Function

**What goes wrong:** If `mix hex.build` fails in the lint job (because `package()` is incomplete or `LICENSE` file is missing from the `files` list), the Release PR gets merged and Hex publish fails in CI.

**Why it happens:** `mix hex.build` validates that all files listed in `package/0 files` exist and that required metadata fields are present.

**How to avoid:** Add `mix hex.build` to the lint job (D-04, already planned). The lint job will catch missing `LICENSE` or bad `package()` metadata before any release can happen.

**Warning signs:** `mix hex.build` exits non-zero in CI with "file not found" or "missing required field".

---

### Pitfall 4: Dependabot Commit Messages Not Picked Up by Release Please

**What goes wrong:** Dependabot creates PRs with commit messages like `chore(deps): bump finch from 0.21.0 to 0.22.0`. If Release Please's changelog sections don't include `chore` type, these updates disappear from the changelog — dependency bumps are invisible.

**Why it happens:** Release Please's default changelog sections omit `chore` type or mark it `hidden: true`.

**How to avoid:** The `release-please-config.json` above includes `{"type": "chore", "section": "Miscellaneous", "hidden": true}` — deps bumps appear in the changelog but are collapsed under "Miscellaneous". If you want deps bumps visible, use `"hidden": false` and add a `deps` section explicitly.

**Warning signs:** Changelog entries for a release don't mention any dependency updates despite Dependabot PRs having been merged.

---

### Pitfall 5: HEX_API_KEY Scope Too Broad

**What goes wrong:** Using a HEX_API_KEY with "write all packages" scope means any compromise of the secret (or the GitHub repo) allows publishing malicious versions of any package under the account.

**Why it happens:** Default Hex key generation grants broad permissions.

**How to avoid:** When generating the HEX_API_KEY on hex.pm, scope it to only the `lattice_stripe` package. GitHub Actions secret should be scoped to the repo (not organization-wide).

**Warning signs:** This is a configuration step at initial setup — no runtime warning. Must be done manually when adding the secret.

```yaml
      - elixir: '1.15'
        otp: '26'
      - elixir: '1.17'
        otp: '27'
      - elixir: '1.19'
        otp: '28'
steps:
  - uses: erlef/setup-beam@v1
    with:
      elixir-version: ${{ matrix.elixir }}
      otp-version: ${{ matrix.otp }}
```

Using `matrix.include` (not a full cartesian product) avoids invalid Elixir/OTP combos (e.g., Elixir 1.15 + OTP 28 is untested). The three locked combos from D-03 map directly to these entries.

**Version strings must be quoted** in YAML (`'1.15'` not `1.15`) to prevent the YAML parser from interpreting them as floats.

## Open Questions

1. **paths-ignore + branch protection deadlock (D-02, Claude's Discretion)**
   - What we know: When all changed files match `paths-ignore`, GitHub skips the workflow entirely and required checks stay "Pending" forever.
   - What's unclear: Whether the maintainer bypass approach (D-36) is sufficient for a solo OSS project, or if the skip job pattern is worth the added complexity.
   - Recommendation: Planner should default to the bypass approach (simpler, acceptable for solo OSS). If the skip job pattern is needed, it requires a dedicated `check-skip` job that always exits 0 and is made the required check instead of individual lint/test jobs.

2. **mix_audit dependency not in mix.exs**
   - What we know: `mix deps.audit` is listed in the lint job (D-04), but `mix_audit` is not in mix.exs `deps()`.
   - What's unclear: Whether Phase 11 should add `{:mix_audit, "~> 2.1", only: :dev, runtime: false}` to mix.exs or whether `mix deps.audit` is available via a different mechanism.
   - Recommendation: Add `mix_audit` to mix.exs in this phase. The planner should include a task for this.

3. **GitHub repo settings (D-43) automation**
   - What we know: Topics, merge strategy, branch deletion, and branch protection require GitHub repo settings that are not file-based.
   - What's unclear: Whether the phase should use `gh repo edit` CLI commands (automatable) or document these as manual steps.
   - Recommendation: Use `gh` CLI where possible (`gh repo edit --squash-merge-commit-title=PR_TITLE`, `gh repo edit --delete-branch-on-merge`, `gh repo edit --topics`). Branch protection requires `gh api` calls. Document all these as a single "configure repo settings" task.
