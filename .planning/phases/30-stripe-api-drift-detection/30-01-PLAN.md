**Step 3 -- Create `LatticeStripe.Drift` module (`lib/lattice_stripe/drift.ex`):**

Module with `@moduledoc false` (internal dev tooling). Public API:

1. `run(opts)` -- Main entry point. `opts` is a keyword list (currently unused, reserved for future flags like `--spec-url`).
   - Calls `fetch_spec/0` to download the spec via Finch (the app is already started by the Mix task).
   - Calls `resource_schemas/1` to extract first-class schemas from the parsed spec.
   - Calls `ObjectTypes.object_map/0` to get the registry.
   - For each entry in `object_map`, calls `known_fields_for/1` to extract `@known_fields` via source parsing.
   - Calls `compare/2` for each registered module to compute additions and removals.
   - Computes new resources (object types in spec but not in `object_map` keys).
   - Returns `{:ok, result_map}` or `{:error, reason}`.

   The result map structure:
   ```elixir
   %{
     drift_count: integer(),        # number of modules with at least one addition
     modules: [                     # list of per-module diffs (only modules with drift)
       %{
         module: atom(),            # e.g. LatticeStripe.Customer
         object_type: String.t(),   # e.g. "customer"
         additions: MapSet.t(),     # fields in spec, not in @known_fields
         removals: MapSet.t()       # fields in @known_fields, not in spec (per D-05)
       }
     ],
     new_resources: [String.t()]    # object types in spec with no registry entry (per D-06)
   }
   ```

2. `format_report(result_map)` -- Formats the result into the human-readable string per D-02:
   ```
   Drift detected in N modules:

   LatticeStripe.Customer (stripe object: "customer")
     + new_field_name (string)
     + another_field (object)
     - removed_field (warning: in @known_fields but not in spec)

   New resources not yet implemented (M):
     tax.calculation
     identity.verification_session
   ```
   For additions, include the field type from the spec if available (simple types: `string`, `integer`, `boolean`, `number`, `array`; for `$ref` or complex types, show `object`). Per D-02 formatting.
   For removals, prefix with `-` and add `(warning: in @known_fields but not in spec)` per D-05.

3. `fetch_spec/0` -- Private. Downloads `https://raw.githubusercontent.com/stripe/openapi/master/openapi/spec3.json` via Finch using the `:lattice_stripe` Finch pool name. Sets `receive_timeout: 30_000`. Returns `{:ok, parsed_map}` or `{:error, reason}`. Uses `Finch.build(:get, url, [], nil)` then `Finch.request(req, LatticeStripe.Finch, receive_timeout: 30_000)`.

   IMPORTANT: Check how Finch is named in the application. Read `lib/lattice_stripe/application.ex` or search for `Finch.start_link` to find the correct pool name. If no Application module exists (this is a library), Finch is started by the user's app. For Mix task context, the task needs to either start Finch inline or use a different HTTP approach. Research found that `Mix.Task.run("app.start")` loads the app but a library has no `Application` module. The solution: start a temporary Finch pool in `fetch_spec/0`:

   ```elixir
   defp fetch_spec do
     finch_name = LatticeStripe.Drift.Finch
     {:ok, _} = Finch.start_link(name: finch_name)
     url = "https://raw.githubusercontent.com/stripe/openapi/master/openapi/spec3.json"

     result =
       :get
       |> Finch.build(url, [], nil)
       |> Finch.request(finch_name, receive_timeout: 30_000)

     case result do
       {:ok, %Finch.Response{status: 200, body: body}} -> Jason.decode(body)
       {:ok, %Finch.Response{status: status}} -> {:error, {:http_error, status}}
       {:error, exception} -> {:error, exception}
     end
   end
   ```

4. `resource_schemas/1` -- Private. Takes the parsed spec map. Filters `components.schemas` for entries where `properties.object.enum` is a single-element list. Returns `%{object_type_string => %{fields: MapSet.t(), types: %{field_name => type_string}}}`. CRITICAL: Use `properties.object.enum[0]` as the key, NOT the schema name (per Pitfall 2 in RESEARCH.md -- e.g., schema `"invoice_item"` has object type `"invoiceitem"`).

   For the `types` map, extract the `"type"` key from each property. If `"type"` is absent but `"$ref"` is present, use `"object"`. If neither, use `"unknown"`.

5. `known_fields_for/1` -- Private. Takes a module atom. Returns `{:ok, MapSet.t()}` or `{:error, reason}`. Uses `module.__info__(:compile)[:source]` to get the source path (returns a charlist -- convert with `List.to_string/1`). Reads the file and applies the regex `~r/@known_fields\s+~w\[([^\]]+)\]/s` to extract field names. Splits on whitespace, creates a `MapSet`. Handle `nil` source path and file read errors gracefully.

6. `compare/2` -- Private. Takes `spec_fields` (MapSet) and `known_fields` (MapSet). Returns `%{additions: MapSet, removals: MapSet}`.

Make `resource_schemas/1`, `known_fields_for/1`, and `compare/2` accessible for testing. Use `@doc false` + `def` (not `defp`) so tests can call them directly. This follows the pattern where `TestClock.Cleanup.parse_duration!/1` is `@doc false` + `def`.
