```elixir
defmodule Mix.Tasks.LatticeStripe.CheckDrift do
  @moduledoc """
  Compares Stripe's published OpenAPI specification against LatticeStripe's
  `@known_fields` module attributes and reports any drift.

  Downloads the latest `spec3.json` from the `stripe/openapi` GitHub
  repository and checks every module registered in
  `LatticeStripe.ObjectTypes` for field additions (in spec, not in
  `@known_fields`) and field removals (in `@known_fields`, not in spec).

  Also reports Stripe object types present in the spec that have no
  corresponding entry in the ObjectTypes registry.

  ## Usage

      # Check for drift (downloads spec on each run):
      mix lattice_stripe.check_drift

  ## Exit codes

  - `0` -- no drift detected; all `@known_fields` match the spec
  - `1` -- drift detected; report printed to stdout
  """

  use Mix.Task

  @shortdoc "Check for Stripe API drift against @known_fields"

  @impl Mix.Task
  def run(args) do
    {opts, _, _} = OptionParser.parse(args, strict: [])
    Mix.Task.run("app.start")

    case LatticeStripe.Drift.run(opts) do
      {:ok, %{drift_count: 0, new_resources: new_resources}}
      when new_resources == [] ->
        Mix.shell().info("No drift detected. @known_fields are up to date.")

      {:ok, result} ->
        Mix.shell().info(LatticeStripe.Drift.format_report(result))
        System.halt(1)

      {:error, reason} ->
        Mix.raise("Drift check failed: #{inspect(reason)}")
    end
  end
end
```

Key details:
- `use Mix.Task` with `@shortdoc` for `mix help` listing
- `@impl Mix.Task` on `run/1`
- `Mix.Task.run("app.start")` BEFORE calling Drift.run -- this loads all modules so `__info__(:compile)` works
- `Mix.shell().info/1` for ALL output (never `IO.puts`)
- `System.halt(1)` for drift-found exit (NOT `Mix.raise` which prints stacktrace) -- per D-02 exit code requirement
- `Mix.raise/1` ONLY for fatal errors (HTTP failure, malformed spec)
- Guard clause `when new_resources == []` ensures that even if there's no per-module drift but there ARE new resources, the report is still shown (but new resources alone do NOT trigger exit code 1 per D-06 -- informational only)

Create `.github/workflows/drift.yml` -- the weekly cron workflow per D-03. Copy the erlef/setup-beam + cache boilerplate from `ci.yml` exactly.

Full workflow content:

```yaml
name: Stripe API Drift Check

on:
  schedule:
    - cron: '0 9 * * 1'  # Mondays at 9:00 UTC
  workflow_dispatch:       # Allow manual trigger from Actions tab

permissions:
  contents: read
  issues: write

jobs:
  drift-check:
    name: Stripe API Drift Check
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4

      - uses: erlef/setup-beam@v1
        with:
          elixir-version: '1.19'
          otp-version: '28'

      - name: Cache deps
        uses: actions/cache@v3
        with:
          path: deps
          key: ${{ runner.os }}-1.19-28-mix-${{ hashFiles('**/mix.lock') }}
          restore-keys: |
            ${{ runner.os }}-1.19-28-mix-

      - name: Cache _build
        uses: actions/cache@v3
        with:
          path: _build
          key: ${{ runner.os }}-1.19-28-build-${{ hashFiles('**/mix.lock') }}
          restore-keys: |
            ${{ runner.os }}-1.19-28-build-

      - name: Install dependencies
        run: mix deps.get

      - name: Compile
        run: mix compile

      - name: Ensure drift issue label exists
        run: |
          gh label create "stripe-drift" \
            --color "#e4e669" \
            --description "Stripe OpenAPI spec fields not yet in @known_fields" \
            --force
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}

      - name: Run drift check
        id: drift
        continue-on-error: true
        run: |
          mix lattice_stripe.check_drift 2>&1 | tee /tmp/drift_report.txt
          echo "exit_code=${PIPESTATUS[0]}" >> $GITHUB_OUTPUT

      - name: Create or update drift issue
        if: steps.drift.outputs.exit_code == '1'
        run: |
          EXISTING=$(gh issue list \
            --label "stripe-drift" \
            --state open \
            --json number \
            --jq '.[0].number // empty')

          BODY=$(cat /tmp/drift_report.txt)
          TITLE="Stripe API drift detected - $(date +%Y-%m-%d)"

          if [ -n "$EXISTING" ]; then
            gh issue comment "$EXISTING" \
              --body "$BODY"
          else
            gh issue create \
              --title "$TITLE" \
              --body "$BODY" \
              --label "stripe-drift"
          fi
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Key implementation details per locked decisions:

1. **D-01 (Spec sourcing):** The Mix task fetches the spec at runtime via Finch. No vendored snapshot. CI caches are for deps/_build only -- the spec itself is downloaded fresh each run by the Mix task.

2. **D-03 (CI notification):** Uses `gh issue create` / `gh issue comment` with `--label "stripe-drift"` for deduplication. The `gh label create --force` step is idempotent -- creates the label on first run, no-ops on subsequent runs. Per Pitfall 4 in RESEARCH.md.

3. **D-03 (Duplicate prevention):** `gh issue list --label "stripe-drift" --state open --json number --jq '.[0].number // empty'` finds any existing open drift issue. If found, adds a comment. If not found, creates a new issue.

4. **Permissions:** `contents: read` + `issues: write` ONLY. No `contents: write` -- the workflow never pushes code. Per security enforcement: GITHUB_TOKEN scoped minimally.

5. **`continue-on-error: true`** on the drift step is required because `System.halt(1)` makes the step fail. We capture the exit code via `PIPESTATUS[0]` and use it in the conditional for issue creation.

6. **`workflow_dispatch`:** Allows manual trigger from the GitHub Actions tab for testing without waiting for Monday cron.

7. **Caching:** Reuses the exact same cache key pattern as `ci.yml` (`${{ runner.os }}-1.19-28-mix-${{ hashFiles('**/mix.lock') }}`). This means deps/build caches are shared between CI and drift workflows -- faster runs.
