## Framing: who, what, why

**Accrue** is the user's higher-level Elixir billing/payments library — analogue of Laravel
Cashier (PHP) and Ruby `pay` gem. It's built on top of LatticeStripe and has its own GSD
planning process (separate repo, separate `.planning/`). Accrue is currently consuming
`lattice_stripe ~> 1.0`.

Accrue's Phase 3 (Core Subscription Lifecycle) is fully unblocked by LatticeStripe 1.0.
Accrue's **Phase 4** (Advanced Billing + Checkout/Portal) is **blocked** on three Stripe
resources that LatticeStripe does not yet expose. Those three resources are the entirety of
v1.1 scope — no speculative work, no adjacent features.

This is the same pattern the project mission memory already describes ("downstream Cashier/Pay
analogue drives priority") but now concrete: the analogue has a name (Accrue), the blockers
have shape, and v1.0 is live so the priority has shifted from "Billing completeness" to
"unblock Accrue's next phase."

## Scope: the Accrue gap list (verbatim)

Pasted as-is from the Accrue session to preserve fidelity. The three resources below are the
full v1.1 scope.

### 1. LatticeStripe.Billing.Meter (new)

Stripe API: `/v1/billing/meters` — https://docs.stripe.com/api/billing/meter
Unblocks: Accrue BILL-11 (metered usage billing) — creating/updating meters is an admin-ish
lifecycle operation, mostly for seeding test and staging envs and for the Accrue admin UI.

**Minimum API surface Accrue will call:**
- `create/3` — create a meter with `display_name`, `event_name`, `default_aggregation.formula`
  (`:sum | :count | :last`), `customer_mapping`, `value_settings`
- `retrieve/3` — fetch by meter id
- `update/4` — update `display_name` and a couple of mutable fields
- `list/3` — list meters for dashboard / admin UI (pagination, `stream/3` follows the
  lattice_stripe convention)
- `deactivate/3`, `reactivate/3` — lifecycle verbs (Stripe exposes these as distinct endpoints,
  not a status update)

**Struct fields Accrue needs to read:**
`id`, `object`, `display_name`, `event_name`, `event_time_window`, `status`
(`:active | :inactive`), `status_transitions`, `default_aggregation.formula`,
`customer_mapping.event_payload_key`, `customer_mapping.type`,
`value_settings.event_payload_key`, `created`, `livemode`

**Notes for the implementer:**
- Meter ids look like `mtr_...`
- Endpoint lives under `/v1/billing/meters/...` — follow the same Resource pattern used in
  `lib/lattice_stripe/subscription.ex`
- Activate/deactivate are POST to `/v1/billing/meters/{id}/deactivate` and `/reactivate` —
  model as dedicated verbs, not status mutations
- No delete; meters can only be deactivated
- No search endpoint

### 2. LatticeStripe.Billing.MeterEvent (new)

Stripe API: `/v1/billing/meter_events` (low-latency "fire and forget") and the higher-throughput
`/v1/billing/meter_event_stream` — https://docs.stripe.com/api/billing/meter_event
Unblocks: Accrue BILL-11 — this is the hot path for reporting usage
(`Accrue.Billing.report_usage/3`). It must be cheap and idempotent.

**Minimum API surface Accrue will call:**
- `create/3` — accepts `event_name`, `payload` (map with at minimum the customer-mapping key
  and a value), `timestamp` (optional; defaults to now), `identifier` (optional but highly
  recommended — used for idempotency across retries)

Accrue does not need to read MeterEvent back — the API returns a thin ack. Keep the struct
minimal: `event_name`, `identifier`, `payload`, `timestamp`, `created`, `livemode`

**Notes for the implementer:**
- This is the single most important one. Accrue will call `create/3` on every usage report, so
  it should:
  - Accept the standard `idempotency_key:` opt (already in client plumbing)
  - Honor the standard `stripe_account:` opt for Connect (Accrue Phase 5)
  - Return `{:ok, %MeterEvent{}} / {:error, %Error{}}` matching the rest of the lib
- Also expose `LatticeStripe.Billing.MeterEventAdjustment.create/3`
  (`/v1/billing/meter_event_adjustments`) — Accrue's dunning-style correction flow will need
  it. **Locked: included in v1.1.**
- The streaming endpoint (`meter_event_stream`) is **deferred** to v1.2+ per locked decision D3.

### 3. LatticeStripe.BillingPortal.Session (new)

Stripe API: `/v1/billing_portal/sessions` — https://docs.stripe.com/api/customer_portal/sessions
Unblocks: Accrue CHKT-02 (Customer Portal) — Accrue creates a portal session and hands the URL
to the host app for redirect. Single call, single return.

**Minimum API surface:**
- `create/3` — required `customer`, optional `return_url`, `configuration`, `locale`, `flow_data`
  (for deep-linking to a specific flow like `subscription_cancel` / `payment_method_update`),
  `on_behalf_of`

**Struct fields Accrue needs to read:**
`id`, `object`, `customer`, `url`, `return_url`, `created`, `livemode`, `locale`,
`configuration`, `flow` (echoed back)

**Notes for the implementer:**
- Sessions are **create-only**. No retrieve, no list, no update, no delete.
- Sibling resource `/v1/billing_portal/configurations` is **deferred** to v1.2+ per locked
  decision D4. For v1.1, hosts manage portal config via the Stripe dashboard.
- Module namespace: `LatticeStripe.BillingPortal.Session` (matches `LatticeStripe.Checkout.Session`).

### What v1.1 explicitly does NOT include

From the Accrue gap audit, everything else in Billing/Connect/Payments that Accrue's roadmap
touches is already in LatticeStripe 1.0. These are **out of scope** for v1.1:
- All existing Billing (Subscription, SubscriptionItem, SubscriptionSchedule, Invoice,
  InvoiceItem, Price, Product, Coupon, PromotionCode)
- All existing Payments (PaymentIntent, SetupIntent, PaymentMethod, Refund, Charge)
- Checkout.Session
- All Connect (Account, AccountLink, LoginLink, Transfer, TransferReversal, Payout, Balance,
  BalanceTransaction)
- Webhook verification (construct_event, generate_test_signature)
- Idempotency plumbing, stripe_account header, stripe_version override

Accrue confirms all of the above work against 1.0 as-is.

### Stripe API version context

Accrue is pinned to `2026-03-25.dahlia` via LatticeStripe's default. All three v1.1 resources
are stable on that version. **No `stripe_version` bump required.**

## Locked decisions

These are final for v1.1. They were made deliberately in conversation before this file was
written. Future planning sessions must not relitigate them without a strong reason.

| ID | Decision | Rationale |
|---|---|---|
| **D1** | Bundle `Billing.Meter` and `Billing.MeterEvent` into a single phase (Phase 20) | They share fixtures, tests, and the new `metering.md` guide. You can't meaningfully exercise MeterEvent without a Meter — they're structurally one feature. |
| **D2** | Include `Billing.MeterEventAdjustment.create/3` in Phase 20's MeterEvent plan | ~30 lines sitting next to MeterEvent. Deferring means re-opening the same file in a later phase for a dunning-correction flow Accrue's roadmap already wants. Tight cohesion > artificial scope limits. |
| **D3** | Defer `/v1/billing/meter_event_stream` (high-throughput variant) to v1.2+ | Accrue explicitly does not need it for Phase 4. Different semantics (streaming, batched, higher cardinality) than the fire-and-forget `meter_events` endpoint. Separate work. |
| **D4** | Defer `BillingPortal.Configuration` CRUDL to v1.2+ | Accrue explicitly does not need it. Session is create-only with a tiny surface; Configuration is a full CRUDL resource with deep UX structs. Adding it triples Phase 21's scope. Hosts manage Portal config via Stripe dashboard in v1.1. |
| **D5** | Phase order: 20 (metering) before 21 (portal) | Metering is higher priority per Accrue's stage ordering (BILL-11 is hot-path, consumed in Accrue's Phase 4 bootstrap), larger scope (more plans), and independent of Portal work. Phase 21 can start as soon as Phase 20 is planned — they do not have a code-level dependency. |

## Architectural note: **no release phase for v1.1**

This is critical and surprising if you haven't traced the git history. **Do NOT add a "Phase 22:
release-please 1.1.0 cut" by analogy with Phase 19.** Here's why:

1. Phase 19 needed a forced release via `"release-as": "1.0.0"` because release-please won't
   promote 0.x → 1.0 on its own when `bump-minor-pre-major: true`. That was a one-shot override.
2. **PR #8 (post-1.0 cleanup, merged as `715a02f`) removed `"release-as"` and flipped
   `bump-minor-pre-major` to `false`**. See `.release-please-manifest.json` (now `1.0.0`) and
   `release-please-config.json` (now has `"packages": { ".": {} }`).
3. Under the new config, a single `feat:` commit on main will auto-bump `1.0.0 → 1.1.0`.
   Release-please opens the PR, merging it triggers tag + Hex publish, all zero-touch. Verified
   in the workflow run after PR #8 merged: `Release Please` ran in 6s, `Publish to Hex.pm` was
   skipped (no version change), no spurious 1.0.0 republish attempt.

**Therefore v1.1 ships automatically the moment Phase 21's last `feat:` commit lands on main.**
The release mechanism is part of the existing CI/CD pipeline, not a milestone phase.

If you feel tempted to add a release phase because "Phase 19 had one," read the CHANGELOG note
above twice. The 0.x → 1.0 graduation was a special case. Normal semver (1.0 → 1.1 → 1.2) is
zero-touch now.

**What about a Highlights narrative for the 1.1 CHANGELOG?** Optional. For a minor release with
two small feature groups, the release-please auto-generated `### Features` list is probably
enough. If you want a narrative, write it as a body paragraph in one of the feat commits (not
a separate staging area); conventional-commit body text lands in the CHANGELOG under the commit
bullet. No staging dance needed.

## Proposed phase structure

Shape mirrors Phase 17/18 (Connect-track feature phases — same problem: new top-level Stripe
resources). Plan counts are estimates; GSD's `/gsd-plan-phase` will finalize them.

### Phase 20 — Billing metering (Meter + MeterEvent + MeterEventAdjustment)

| Plan | Scope |
|---|---|
| 20-01 | **Wave 0 bootstrap** — canonical `Meter`, `MeterEvent`, `MeterEventAdjustment` fixtures; `stripe-mock` probe to confirm all three endpoints behave as documented; new `test/support/fixtures/metering.ex` |
| 20-02 | **Nested structs** — `Meter.DefaultAggregation`, `Meter.CustomerMapping`, `Meter.ValueSettings`, `Meter.StatusTransitions`. Follows the Phase 17 nested-struct pattern (typed struct + `from_map/1` + `@known_fields` + `extra`). |
| 20-03 | **`LatticeStripe.Billing.Meter` resource module** — CRUDL (`create/3`, `retrieve/3`, `update/4`, `list/3`, `stream/3`) + lifecycle verbs `deactivate/3` and `reactivate/3`. Bang variants for all. No delete. No search. |
| 20-04 | **`LatticeStripe.Billing.MeterEvent` + `MeterEventAdjustment`** — create-only for both. Idempotency key + `stripe_account:` opts threaded through. Minimal structs. |
| 20-05 | **Integration tests** — full lifecycle against `stripe-mock`: seed a meter, report events through it, adjust, deactivate, list filtered by status. Mirrors Phase 17's integration-test plan shape. |
| 20-06 | **Guide + ExDoc** — new `guides/metering.md` (usage reporting idiom, reconciliation notes, webhook handoff pointers), new ExDoc group `Billing Metering` in `mix.exs:groups_for_modules`, cross-links from existing `guides/billing.md` or equivalent. |

### Phase 21 — BillingPortal.Session

| Plan | Scope |
|---|---|
| 21-01 | **Wave 0 bootstrap** — canonical `Session` fixture, `stripe-mock` probe for `/v1/billing_portal/sessions` |
| 21-02 | **Nested structs** — `Session.FlowData` (for deep-link flows: `subscription_cancel`, `payment_method_update`, etc.) |
| 21-03 | **`LatticeStripe.BillingPortal.Session` resource module** — `create/3` + `create!/3` only. No retrieve/list/update/delete. Minimal struct with `url`, `customer`, `return_url`, `flow`, etc. |
| 21-04 | **Integration test + guide** — full portal flow against `stripe-mock`; extend or create `guides/customer-portal.md` with Accrue-style usage example. Short plan, maybe bundled with 21-03. |

**Total estimated scope for v1.1:** ~10 plans across 2 phases. Significantly smaller than Phase
17 (6 plans) or Phase 18 (9 plans). Realistic single-session milestone.
