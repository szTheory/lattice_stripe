**FormEncoder edge cases (add to test/lattice_stripe/form_encoder_test.exs):**

Add a new `describe "encode/1 edge cases"` block with tests for:
1. Empty list value: `%{items: []}` -- should encode to `""` or appropriate empty encoding (verify actual behavior)
2. Empty nested map value: `%{metadata: %{}}` -- should encode to `""` or be omitted (verify actual behavior)
3. Mixed string and atom keys: `%{"string_key" => "val1", atom_key: "val2"}` -- both should encode correctly
4. Unicode values: `%{name: "Rene"}` with accented characters and `%{name: "Tokyo"}` with CJK -- should URL-encode correctly
5. Value containing equals sign: `%{query: "a=b"}` -- equals must be encoded
6. Value containing ampersand in nested context: `%{metadata: %{note: "a&b"}}` -- ampersand must be encoded in nested bracket notation
7. Deeply nested (4+ levels): `%{a: %{b: %{c: %{d: "deep"}}}}` -- should produce `a[b][c][d]=deep`
8. Array of empty maps: `%{items: [%{}, %{}]}` -- verify behavior (likely empty string or omitted)
9. Nil in array: `%{items: ["a", nil, "b"]}` -- verify nil is skipped in array encoding
10. Integer zero and negative values: `%{amount: 0}` and `%{adjustment: -500}` -- should encode as "0" and "-500"

**Error normalization edge cases (add to test/lattice_stripe/error_test.exs):**

Add a new `describe "Error.from_response/3 unusual shapes"` block with tests for:
1. Missing "error" key entirely: `%{"message" => "plain text error"}` -- should fall back to :api_error
2. Empty error map: `%{"error" => %{}}` -- should produce :api_error with nil message
3. Null/nil message: `%{"error" => %{"type" => "api_error", "message" => nil}}` -- message should be nil
4. Error body is a string (not map): status 502, body as a string `"Bad Gateway"` -- verify from_response/3 handles gracefully (it expects a map, so test what happens with non-map input if the caller pre-parses, or test the error struct for the 502 fallback case)
5. Extra unknown fields in error: `%{"error" => %{"type" => "card_error", "message" => "Declined", "unknown_field" => "value", "nested" => %{"deep" => true}}}` -- unknown fields should not crash, error struct should still parse correctly
6. Type is nil: `%{"error" => %{"type" => nil, "message" => "Error"}}` -- should fall back to :api_error
7. Very long message string (1000+ chars): verify it's stored without truncation
8. Numeric status edge cases: status 0, status 999 -- from_response/3 should still produce an Error struct

**Pagination cursor edge cases (add to test/lattice_stripe/list_test.exs):**

Add a new `describe "from_json/1 cursor edge cases"` block with tests for:
1. Items with string ID values containing special characters: `%{"id" => "cus_abc/def"}` -- _last_id should preserve the value
2. Items with integer ID (edge case): `%{"id" => 12345}` -- verify _last_id behavior
3. Mixed items where some have id and some don't: `[%{"id" => "cus_1"}, %{"name" => "no_id"}, %{"id" => "cus_3"}]` -- _first_id should be "cus_1", _last_id should be "cus_3" or nil depending on last item
4. Single item with nil id: `[%{"id" => nil}]` -- verify _first_id and _last_id behavior
5. Very large data array (100 items): verify _first_id is first item's id and _last_id is last item's id (no off-by-one)

For each test, read the production code first to understand what the ACTUAL behavior is, then write the test to assert that behavior. If the actual behavior seems wrong (e.g., crashes on edge case), document it with a comment but still write the test asserting the current behavior.
