       ```elixir
       @doc """
       Creates a lazy stream that auto-paginates through all items matching the request.

       Makes the initial API call, then lazily fetches subsequent pages as the stream
       is consumed. Emits individual items (flattened from pages), not page structs.

       Raises `LatticeStripe.Error` if any page fetch fails (after retries are exhausted).

       ## Example

           req = %LatticeStripe.Request{method: :get, path: "/v1/customers", params: %{"limit" => "100"}}
           client
           |> LatticeStripe.List.stream!(req)
           |> Stream.take(50)
           |> Enum.to_list()

       ## Memory Warning

       Without `Stream.take/2`, the stream will fetch ALL matching objects.
       For large collections, always limit with `Stream.take/2`.
       """
       @spec stream!(Client.t(), Request.t()) :: Enumerable.t()
       def stream!(%Client{} = client, %Request{} = req) do
         Stream.resource(
           fn -> fetch_page!(client, req) end,
           &next_item/1,
           fn _acc -> :ok end
         )
       end
       ```

    4. Implement `next_item/2` — the core state machine (accumulator is a `%List{}`):
       ```elixir
       defp next_item(%__MODULE__{data: [], has_more: false}, _client) do
         {:halt, :done}
       end

       defp next_item(%__MODULE__{data: [], has_more: true} = list, client) do
         next_page = fetch_next_page!(list, client)
         next_item(next_page, client)
       end

       defp next_item(%__MODULE__{data: [item | rest]} = list, _client) do
         {[item], %{list | data: rest}}
       end
       ```

       NOTE: When buffer empties and has_more is true, immediately fetch next page and recurse into next_item to start draining the new buffer. This avoids emitting an empty list `{[], next_page}` which would cause Stream.resource to call next_fun again anyway.

       ```elixir
       defp build_next_page_request(%__MODULE__{} = list) do
         base_params = Map.drop(list._params, ["starting_after", "ending_before", "page"])

         pagination_params =
           cond do
             # Search pagination (D-16): use page token
             list.object == "search_result" && list.next_page ->
               %{"page" => list.next_page}

             # Backward cursor pagination (D-06): use first item ID
             Map.has_key?(list._params, "ending_before") && list.data != [] ->
               %{"ending_before" => get_item_id(List.first(list.data))}

             # Forward cursor pagination: use last item ID
             list.data != [] ->
               %{"starting_after" => get_item_id(List.last(list.data))}

             true ->
               %{}
           end

         # Strip idempotency_key from opts — page fetches are GET (D-31)
         opts = Keyword.delete(list._opts, :idempotency_key)

         %Request{
           method: :get,
           path: list.url,
           params: Map.merge(base_params, pagination_params),
           opts: opts
         }
       end

       defp get_item_id(item) when is_map(item), do: item["id"]
       ```

       NOTE on backward pagination cursor (Pitfall 6): For `ending_before`, the cursor is the FIRST item's ID (Stripe returns items before that ID). For `starting_after`, the cursor is the LAST item's ID. `list.data` at this point may have been partially consumed by the stream, so we need to be careful. Actually, when we reach `data: []` and need to fetch next page, all items have been consumed. We need to remember the boundary item ID.

       CORRECTION: When `next_item/2` hits `data: []` and `has_more: true`, the data buffer is empty — we've already emitted all items. We need the cursor from the ORIGINAL page, not the consumed buffer. The fix: store the boundary cursor on the List struct itself, OR extract it before draining.

       Better approach: extract cursors from the data list BEFORE it gets consumed. In `fetch_page!` or in `next_item`, when we receive a fresh page, compute and store cursors. Add `_first_id` and `_last_id` fields to the List struct (private, underscore-prefixed). OR compute them in build_next_page_request from the full page's data before draining.

       Actually, the simplest and most correct approach: store `_last_id` and `_first_id` when building the List from JSON (in from_json), since at that point `data` is complete. Add two internal fields:

       In `list.ex` defstruct, add: `_first_id: nil, _last_id: nil`

       In `from_json/3`, after building the struct, compute:
       ```elixir
       items = decoded["data"] || []
       first_id = case items do
         [first | _] when is_map(first) -> first["id"]
         _ -> nil
       end
       last_id = case List.last(items) do  # Elixir's List.last, not our List
         item when is_map(item) -> item["id"]
         _ -> nil
       end
       ```

       Then `build_next_page_request` uses `list._last_id` for forward and `list._first_id` for backward.

       NOTE: Use `Elixir.List.first/1` and `Elixir.List.last/1` to avoid conflict with the module's own name. Or use pattern matching: `[first | _] = items` and `items |> Enum.at(-1)`.
