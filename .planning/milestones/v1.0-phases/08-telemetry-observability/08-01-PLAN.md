**Public functions:**

1. `request_span(client, req, idempotency_key, fun)` — `@doc false`. If `client.telemetry_enabled`, wraps `fun` in `:telemetry.span(@request_event, start_meta, ...)`. The span function calls `fun.()` which returns `{result, attempts}`, then builds stop metadata. If telemetry disabled, calls `fun.()` directly and extracts result. Returns the result (not the tuple).

2. `emit_retry(client, method, url, error, attempt, delay_ms)` — `@doc false`. If `client.telemetry_enabled`, calls `:telemetry.execute(@retry_event, %{attempt: attempt, delay_ms: delay_ms}, %{method: method, path: extract_path(url), error_type: error.type, status: error.status})`.

3. `attach_default_logger(opts \\ [])` — Public, documented. Stub the function signature and `@doc` now, but implementation will be completed in Plan 02. For now: `def attach_default_logger(_opts \\ []), do: :ok`.

4. `webhook_verify_span(opts \\ [], fun)` — `@doc false`. Stub for Plan 02. For now: `def webhook_verify_span(_opts \\ [], fun), do: fun.()`.

**Private helpers (extracted from Client):**

1. `defp build_start_metadata(client, req)` — Builds: `%{method: req.method, path: req.path, resource: resource, operation: operation, api_version: client.api_version, stripe_account: client.stripe_account}` where resource and operation come from `parse_resource_and_operation/2`.

2. `defp build_stop_metadata(result, idempotency_key, attempts, start_meta)` — Three clauses matching the existing `telemetry_stop_metadata` pattern in Client, but ALSO merging in all start_meta fields (method, path, resource, operation, api_version, stripe_account) so stop event has full context (RESEARCH Pitfall 2). Clauses:
   - `{:ok, %Response{}} ->` status: :ok, http_status, request_id, attempts, retries
   - `{:error, %Error{type: :connection_error}} ->` status: :error, error_type: :connection_error, idempotency_key, attempts, retries
   - `{:error, %Error{}} ->` status: :error, http_status, error_type, request_id, idempotency_key, attempts, retries

3. `defp extract_path(url)` — Moved from Client. `URI.parse(url).path` with fallback to `url`.

4. `defp parse_resource_and_operation(method, path)` — (per D-05) Parse URL path to extract resource name and operation. Implementation:
   - Strip `/v1/` prefix, split on `/`
   - Pattern match on segments:
     - `[resource_plural]` — POST = "create", GET = "list"
     - `[resource_plural, id]` — GET = "retrieve", POST = "update", DELETE = "delete"
     - `[resource_plural, id, action]` — operation = action (e.g., "confirm", "capture", "cancel", "expire")
     - `[namespace, resource_plural]` — POST = "create", GET = "list", resource = "namespace.singular" (checkout.session)
     - `[namespace, resource_plural, id]` — retrieve/update/delete, resource = "namespace.singular"
     - `[namespace, resource_plural, id, action]` — operation = action
     - `[resource_plural, "search"]` — operation = "search"
     - `[namespace, resource_plural, "search"]` — operation = "search"
     - Fallback: `{path, to_string(method)}`
   - `singularize/1` helper: strip trailing "s" for regular plurals (customers->customer, sessions->session, refunds->refund). Handle "payment_intents"->"payment_intent", "setup_intents"->"setup_intent", "payment_methods"->"payment_method", "checkout" stays as namespace.
   - ID detection: segments matching `~r/^(cus|pi|seti|pm|re|cs|evt|ch)_/` or any segment with `_` prefix are IDs. Also any alphanumeric segment longer than 10 chars that isn't a known resource/action name.

**Aliases needed:** `alias LatticeStripe.{Error, Response}` and `require Logger`.
